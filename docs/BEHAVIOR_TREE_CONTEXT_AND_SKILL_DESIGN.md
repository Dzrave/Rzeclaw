# 行为树与状态机：上下文与 Skill 设计讨论

本文档针对 **`行为树与状态机下上下文与skill相关`** 中的议题，做**可行性、科学性**分析，并在可行前提下讨论**更佳做法与实现方案**。优先讨论，不做具体实现；若后续采纳，再并入主设计文档并拆工单。

---

## 一、文档要点回顾

讨论文档主要包含：

| 议题 | 内容摘要 |
|------|----------|
| **ECS 与黑板** | 用 ECS（Entity-Component-System）管理 AI 状态；Entity=会话/任务，Component=上下文组件化（如 UserIntentComponent、ErrorLogComponent），System=BT/FSM；进化出新技能时定义新 Component。 |
| **本地小模型作 FSM 仲裁** | 本地小模型只输出严格 JSON/枚举（如 STATE_IDLE、ROUTE_TO_LOCAL_SCRIPT、ESCALATE_TO_CLOUD），零成本高频轮询，仅 ESCALATE 时唤醒云端。 |
| **进化安全** | 双重生成：功能代码 + 单元测试；沙盒执行测试，通过后才 merge 到主行为树库。 |
| **拓扑自我迭代** | BT 整树为 JSON；失败日志 + BT JSON 交给 LLM 重构（装饰器、Selector/Sequence 层级），数据驱动的节点级优化。 |
| **记忆衰减** | 行为树节点带 Usage_Frequency；冷门脚本降级或移出优先匹配队列。 |
| **去中心化上下文** | 全局黑板极轻量；**节点级记忆**：每个 Action 节点自带小型记忆库，只加载与该节点相关的上下文，执行完只向黑板返回干练结果并清空节点内过程记忆；**Token 隔离**。 |
| **Skill 本质** | Skill = 已固化的行为树子树（预制体）；生命周期：探索期（LLM 试错）→ 提炼期（进化）→ 成熟期（Skill JSON）→ 调用期（0 Token 本能）。 |
| **技能胶囊** | Skill 自带专属短期记忆（如过去 3 天修改的类名/函数名）；触发时**自带上下文**进入工作，其他模块不感知。 |

## 二、可行性及科学性分析

### 2.1 去中心化上下文（节点级记忆）

**结论：可行且科学。**

- **科学性**：认知上符合「工作记忆局部性」——执行特定任务时只需与该任务相关的上下文，全局堆砌长文本会导致注意力涣散与幻觉。将「中心化长 prompt」拆成「轻量全局黑板 + 节点级局部上下文」，与当前多智能体与 RAG 中「按需加载、用毕压缩」的实践一致。
- **可行性**：技术上可实现。每个节点（或每棵子 flow）拥有可选「局部上下文」：执行前从专属存储或黑板命名空间加载，执行后只将摘要/结果写回黑板，过程细节不进入全局会话。与现有主设计中的**黑板**（§12.2）自然衔接：黑板已约定「全局轻量、槽位可命名」；节点级记忆可视为**黑板按 flowId/nodeId 命名空间划分**，或节点专属小型存储（如小向量库、键值缓存）。

**与主设计的关系**：主设计已有「黑板 + 仅向黑板返回干练结果」；尚未明确「每个节点自带可加载/可清空的局部上下文」。将节点级记忆定义为**黑板的名字空间子集**或**节点级 context 槽**即可纳入，无需另建一套记忆系统。

---

### 2.2 Skill 的本质：固化行为树链路

**结论：与当前设计高度一致；可显式统一为「Skill = 脚本或 flow」双形态。**

- **科学性**：将 Skill 定义为「已验证、可复用的执行链路」符合「从探索到本能」的进化叙事；在游戏 AI 与机器人中，技能 = 预制子行为树是成熟做法。
- **可行性**：项目中已有两类可复用能力：  
  - **Skill**（`src/skills/`）：名称 + 描述 + inputSchema + scriptPath，由 getMergedTools 转为工具，供 Agent 或工具链调用。  
  - **Flow**（BT/FSM）：由路由匹配后执行，或通过 FSM 的 runFlow、BT 的 FSM 节点互相调用。  
  讨论文档中的「Skill = 行为树子树」在现有设计中即 **flow 作为可调用子流程**；「探索 → 提炼 → 成熟 → 调用」已对应「runAgentLoop 试错 → 轨迹/LLM 生成 flow → 入库 → 路由/runFlow 执行」。

**更佳做法**：  
- 在**概念层**显式统一：**Skill 有两种形态**——(1) **脚本型**：当前 Skill（脚本 + schema），暴露为 Tool；(2) **流程型**：即 flow（BT/FSM），可作为子 flow 被其他 flow 或路由调用。二者在「可复用、可进化、0 Token 调用」上一致，仅在实现上分别为「单工具」与「子树/子状态机」。  
- 在**路由与执行层**：保持现有「路由匹配 flow」「FSM 内 runFlow」「BT 内 FSM 节点」；可选地允许「Skill 描述符」指向 flowId，使「技能」目录与「流程库」在配置或元数据上可统一管理（例如某 Skill 条目的 scriptPath 为空而 flowId 有值，则执行时跑 flow）。

### 2.3 技能胶囊（Skill 自带专属记忆）

**结论：可行且科学；建议实现为「flow/节点级上下文」的特例。**

- **科学性**：不同技能关注不同领域（如「代码审查」关注近期修改的符号，「部署」关注环境与版本）；为技能绑定专属短期记忆可减少无关信息干扰、提高判断准确性，与「节点级记忆」同属分布式上下文思路。
- **可行性**：实现上可复用「节点级记忆」机制：为某 flow 或某节点配置「contextScope」或「localMemoryId」，执行时加载该 scope 下的缓存（如最近 N 次执行的相关实体、或过去 K 天的某类记录）；执行完毕将本次结果摘要写回该 scope，可选清空过程细节。与主设计中的**黑板**结合：黑板槽可按 `flowId` 或 `skillId` 命名（如 `skill_code_review.lastModifiedSymbols`），实现「技能自带上下文」而不必新建一套记忆系统。

**更佳做法**：  
- 不单独建「Skill 记忆系统」，而是将**技能胶囊**定义为：**flow 或节点上可选的「专属上下文槽」**。  
- 槽的数据来源可配置：例如「最近 3 天某目录下的修改记录」「该 flow 上次执行的结果摘要」等，由 BT/FSM 执行器在进入该 flow/节点时加载，退出时按策略保留摘要并清理。  
- 与现有记忆系统（L1/L2、session summary）的关系：技能专属槽可**只写不读**中心记忆，或仅将「结论」写回 L1（可选），避免技能内部细节污染全局记忆。

### 2.4 ECS 范式管理 AI 状态

**结论：科学但非必要；可作为远期结构化选项。**

- **科学性**：ECS 将数据（Component）与逻辑（System）解耦，适合多系统查询「拥有某 Component 的 Entity」；在游戏引擎中验证充分。用于 AI 状态时，Session 为 Entity，UserIntent、LocalFileContext、ErrorLog 等为 Component，BT/FSM 为 System，进化出新技能即新增 Component 类型，逻辑清晰。
- **可行性**：可行，但引入实体/组件/系统的抽象与查询层，实现成本高于当前「session + 黑板 key-value」。  
- **更佳做法**：  
  - **短期**：维持**扁平黑板 + 命名空间**（如 `blackboard[flowId.nodeId.result]`）；已能表达「节点级/技能级」上下文，且与 §12.2 一致。  
  - **远期**：若出现「多系统按 Component 类型筛选 Entity」的需求（如 Heartbeat 只处理带 ChecklistComponent 的会话），再引入 ECS 或类 ECS 的轻量抽象（Entity 表 + Component 表 + 按类型查询），避免过早复杂化。

---

### 2.5 本地小模型作 FSM 条件仲裁者

**结论：可行且与主设计已对齐；可强化「结构化输出」约定。**

- **科学性**：用本地小模型做意图/状态分类，只输出枚举或固定 JSON，成本低、延迟小，适合高频轮询；仅当分类为「需云端」时才调用大模型，符合分层控制思路。
- **可行性**：主设计已含**可选 intentClassifier**（§4.1、§10）；讨论文档强调的「只输出严格 JSON/Enum」是很好的实现约束，可降低解析错误与滥用。
- **更佳做法**：在配置或接口中明确 **intentClassifier 的输出 schema**（如 `{ "state": "ROUTE_TO_LOCAL_SCRIPT" | "ESCALATE_TO_CLOUD", "flowId?: string", "confidence": number }`），Gateway 或会话级 FSM 按 state 迁移；并约定仅当 confidence ≥ 阈值且 state 为 ROUTE_TO_LOCAL_SCRIPT 时走 Executor，否则走 Deep_Reasoning。

### 2.6 进化安全：沙盒试飞与双重生成

**结论：强烈建议采纳；与 §12.3 进化插入树配套。**

- **科学性**：LLM 生成代码直接入库存在崩溃、越权、破坏性操作等风险；「功能代码 + 单元测试」双重生成并在沙盒中先跑测试，通过再 merge，是标准的测试驱动与安全边界实践。
- **可行性**：可实现。进化流程（§12.3）在「生成脚本 + 节点 JSON」之后增加：生成**单元测试代码**；在沙盒（受限进程/Docker/权限限制）中执行测试；仅当全部通过才注册 Tool 并写入/插入树；失败则将错误回传 LLM 修正（可选 Retry 上限）。
- **更佳做法**：将「双重生成 + 沙盒测试」写入主设计 §12.3 的**实现要点**，并单独工单（如 WO-BT-025）：进化产出必须含测试；沙盒环境与通过标准可配置；审计记录「进化尝试/通过/拒绝」。

---

### 2.7 拓扑结构的自我迭代（LLM 重构 BT）

**结论：可行但成本高；建议作为可选进阶。**

- **科学性**：让 LLM 根据失败日志与当前 BT JSON 调整拓扑（如增加重试装饰器、调整 Selector 顺序），属于「元层优化」，与「失败分支标记与替换」（WO-BT-018）同族但更全局。
- **可行性**：需定义「何时触发」「输入给 LLM 的上下文边界」「输出为整棵 BT JSON 的校验与版本管理」；实现成本高，且错误修改可能破坏已有稳定 flow。
- **更佳做法**：先做**局部**替换（WO-BT-018：单分支替换），再在需求明确时做**全局**拓扑优化：触发条件（如某 flow 失败率持续高于阈值）、输入（该 flow JSON + 近期失败案例摘要）、输出（新 JSON 进入待审核分支，A/B 或人工确认后替换）。

---

### 2.8 记忆衰减与使用频率

**结论：可行且与经验迭代自然结合。**

- **科学性**：冷门本能常驻会拖累意图匹配与维护成本；按使用频率降级或移出优先队列，符合「资源有限下的注意力分配」。
- **可行性**：主设计已有 flow 元数据（successCount、failCount、lastUsed）；扩展 **usageFrequency** 或「最近 N 次调用时间」即可；路由优选时在「成功率」基础上加入「近期使用频率」权重，冷门 flow 自动靠后。
- **更佳做法**：在 Phase D 经验迭代中增加可选策略：如 `preferRecentHighUsage`；或定义「冷门阈值」（如 30 天未用），冷门 flow 仍可匹配但排在同意图候选末尾，或仅在使用率统计中参与，不参与默认优选。无需单独「删除」flow，避免误删仍有用的长尾能力。

## 三、与主设计的衔接建议（仅设计层面）

| 文档议题 | 主设计对应 | 建议 |
|----------|------------|------|
| 去中心化上下文 / 节点级记忆 | §12.2 黑板 | 将「节点级/flow 级局部上下文」定义为黑板命名空间或节点 context 槽；执行前加载、执行后摘要写回并清空过程细节。 |
| Skill = 固化链路 | Flow（BT/FSM）+ 现有 Skill（脚本） | 显式统一为「Skill 双形态」：脚本型 Tool 与流程型 flow；可选 Skill 元数据指向 flowId。 |
| 技能胶囊（Skill 自带记忆） | 节点级记忆 + 黑板 | flow/节点可选「专属上下文槽」，数据来源可配置；与黑板按 flowId/skillId 命名一致。 |

| 本地小模型 FSM 仲裁 | §4.1 intentClassifier、§12.1 会话级 FSM | 明确 classifier 输出 schema（state enum + confidence）；与状态迁移表结合。 |
| 进化安全（双重生成+沙盒） | §12.3 进化插入树 | 纳入 §12.3 实现要点；单独工单（如 WO-BT-025）。 |

| 记忆衰减 | Phase D 经验迭代、flow meta | 扩展 meta 为含 usageFrequency/最近使用；路由优选时支持「冷门靠后」策略。 |

## 四、总结

- **可行性**：讨论文档中的**去中心化上下文、节点级记忆、Skill 为固化链路、技能胶囊、本地小模型仲裁、进化安全、记忆衰减**均可行，且与主设计中的黑板、flow、Skill、经验迭代、会话级 FSM、intentClassifier 可自然衔接。**ECS** 与 **拓扑自我迭代** 可选、成本较高，适合作为远期或进阶。
- **科学性**：分布式上下文与 Token 隔离、Skill 生命周期、沙盒测试驱动进化、使用频率衰减均符合既有工程与认知实践。
- **更佳做法**：  
  - 节点级/技能级记忆**统一到黑板命名空间或节点 context 槽**，不另建一套记忆系统。  
  - **Skill 显式双形态**（脚本型 + 流程型），与 flow 库和路由统一管理。  
  - 进化安全**强制双重生成 + 沙盒测试**，并写入主设计工单。  
  - 记忆衰减作为**经验迭代的扩展策略**（冷门靠后），不删除 flow。

后续若落实，可将「节点级/flow 级上下文与技能胶囊」并入主设计 §12.2（黑板）或新增小节，并拆为 Phase F 之后工单；进化安全并入 WO-BT-024 或单列 WO-BT-025；记忆衰减并入 Phase D 配置与路由逻辑。
