# 记忆折叠（5 天滑动情景记忆）— 项目纳入与必要性评估

本文档基于 **`docs/记忆折叠构建讨论`** 的讨论内容，对「5 天滑动情景记忆 + 睡前折叠」机制进行**项目现状对照、必要性评估与纳入建议**。仅做评估与设计级建议，不包含实施工单。

---

## 一、讨论内容摘要

### 1.1 问题陈述

- **现象**：智能体在 24 小时某凌晨重置后，仿佛「忘记」前一天发生的事。
- **根因**：传统 RAG（向量库）只有相似度检索，**无时间连贯性**；问「昨天下午卡在哪个 Bug」可能返回数月前相似 Bug，导致「每日重置即失忆」的体验。
- **诉求**：接受「约 5 天记忆保留、动态清理更新」，使记忆**连续、可持续、可准确引用近期进展**。

### 1.2 讨论提出的方案：三级记忆 + 睡前折叠

| 层级 | 名称 | 说明 | 生命周期 |
|------|------|------|----------|
| 1 | **瞬时记忆（Raw Context）** | 当前对话原话（如最近 20 轮） | 凌晨重置时清空 |
| 2 | **5 天滑动情景记忆（Rolling Episodic）** | 高密度、结构化的「时间轴账本」，常驻 System Prompt | 按日滑动，FIFO 淘汰 |
| 3 | **永久知识库（Long-term RAG）** | 被 5 天窗口淘汰的内容，若有价值则提炼入向量库 | 长期保留 |

核心机制：

- **睡前折叠（夜间压缩）**：凌晨重置时，由**架构师 Agent** 将「当日全部对话」提炼为约 100 字以内的**日级摘要** + **Pending Tasks**，插入到 5 天账本的「Yesterday (-1)」位。
- **滑动淘汰**：原 Day -1→-2、…、Day -5 被挤出；被挤出的 Day -5 经判定：有持久价值则打入 RAG，否则丢弃（GC）。
- **无感注入（清晨唤醒）**：用户当日首次发消息时，Gateway 将 5 天账本转为精简自然语言，**注入系统提示词**，使模型能接上「昨天/前天的进展」。

### 1.3 提议的数据结构（时间衰减账本）

```json
{
  "memory_window": "5_days",
  "current_focus": "正在重构战斗结算代码，遇到装甲穿透数值不平滑问题。",
  "rolling_ledger": [
    { "day": "Yesterday (-1)", "date": "2026-03-14", "summary": "...", "pending_tasks": ["..."] },
    { "day": "Day -2", "date": "2026-03-13", "summary": "..." },
    …
  ]
}
```

## 二、与当前实现的对照

### 2.1 已有能力（可直接复用或扩展）

| 能力 | 当前实现位置 | 与「记忆折叠」的关系 |
|------|----------------|----------------------|
| **会话内摘要（L0）** | `write-pipeline.ts`: `generateL0Summary` | 每 N 轮生成摘要，写入 `sessionSummary`；可视为「当日瞬时记忆」的压缩来源之一。 |
| **会话级总结写入文件** | `session-summary-file.ts`: `writeSessionSummaryFile` | 按 session 写 `.rzeclaw/session_summaries/{sessionId}.md`；可与「当日多会话」合并作为折叠输入。 |
| **昨日小记（读）** | `session-summary-file.ts`: `readYesterdaySummary` | 按 **文件 mtime 为昨天** 的 session 摘要拼接，供 `memory.yesterdaySummary` RPC；**未**注入 Agent 系统提示。 |
| **系统提示中的「前文」** | `agent/loop.ts` + `sessionSummary` | 当前仅注入**单段** `[Previous context summary]`（来自当前会话的 L0 摘要），**无多日时间轴**。 |
| **复盘 / 架构师** | `retrospective/architect.ts`: `runRetrospective` | 读遥测、产出补丁（flow_edit / motivation_merge / report），写 pending；**未**做「当日对话 → 日级摘要 + pending_tasks」的折叠。 |
| **L1 写入与 RAG 上游** | `write-pipeline.ts`: `flushToL1` | 摘要与事实写入 store（后续可 promote 到 L2/向量）；被淘汰的 Day -5 可经此路径「有价值则入 RAG」。 |

### 2.2 缺口（要实现讨论方案尚缺的部分）

| 缺口 | 说明 |
|------|------|
| **5 天滚动账本存储** | 无 `rolling_ledger` 形态的 JSON（按日队列、current_focus、pending_tasks）；需新增存储路径与 schema。 |
| **夜间「折叠」流水线** | 架构师当前只消费**遥测**，不消费「当日对话原文」；需在凌晨（或触发点）拉取当日已发生的会话/消息，调用 LLM 生成「日级摘要 + pending_tasks」，并执行 ledger 的 prepend + FIFO 淘汰。 |
| **淘汰日 → RAG 判定** | Day -5 被挤出时，需判定是否写入长期 RAG（如调用现有 L1/store 或直接写内源 RAG）；讨论中由架构师「瞥一眼」决定，需落实为规则或轻量模型。 |
| **清晨 / 请求时注入** | Gateway 或 chat-executor 在构造 Agent 上下文时，需**读取 5 天账本**，转为自然语言并拼接到 system prompt，且与现有 `sessionSummary` 协同（例如「5 天梗概 + 当前会话摘要」）。 |
| **「当日对话」数据来源** | 折叠需要「今天发生过的对话」；当前会话在内存/快照中，若凌晨重置则会话可能已清空。需约定：要么在重置前持久化「当日消息」供折叠消费，要么在每次会话结束/flushToL1 时增量追加到「今日缓冲」，午夜对缓冲做一次压缩入账本。 |

### 2.3 逻辑一致性要点

- **时间边界**：「今日」需与「凌晨重置」对齐，建议用**本地日历日**（或配置的 timezone），以便「昨日」与「Day -1」一致。
- **多会话**：同一 workspace 下可能多 session；账本可以是 **per-workspace**（所有会话合并为「今日」一条摘要），或 per-session（每个 session 一条日摘要，注入时按当前 session 过滤）。讨论未明确，设计时需二选一或支持配置。
- **与复盘的衔接**：折叠产出的 `pending_tasks` 可与现有「待审区 / 早报」结合（例如作为早报中的「昨日未完成任务」），避免两套「待办」并行。

## 三、必要性评估

### 3.1 问题是否成立

- **成立**：若存在「凌晨重置」或「跨日断会话」的用法，且用户期望「接着昨天的进度继续」，当前仅靠 RAG + 单段 sessionSummary 确实容易**时间错位**或**上下文丢失**，与讨论中的痛点一致。
- **前提**：项目是否真的在「凌晨」做会话清空/重置？若当前仅是「会话超时或关闭后下次无状态」，则「昨日」的获取已部分由 `readYesterdaySummary`（昨日 session 文件）覆盖，但**未注入 Agent**，体验上仍可能像「忘了昨天」。

### 3.2 价值与收益

| 维度 | 收益 |
|------|------|
| **体验** | 消除「跨日失忆」感；用户说「继续昨天的」时，模型能基于 5 天账本准确接上。 |
| **Token** | 用约 300–500 token 的 5 天高密度摘要，替代「拉取多日原始对话」的高消耗。 |
| **时间一致性** | 账本按时间顺序排列，避免 RAG 把「3 天前」与「昨天」混淆。 |
| **与现有架构兼容** | 复用 L0/会话摘要、复盘架构师、L1/RAG 写入；新增「账本 + 折叠 + 注入」三条线即可。 |

### 3.3 成本与风险

| 维度 | 说明 |
|------|------|
| **实现成本** | 需新增：账本存储与读写、折叠流水线（含「当日数据」来源）、注入点与 prompt 拼接、淘汰→RAG 逻辑；与复盘/遥测的触发时机需协调（如复盘 cron 与折叠 cron）。 |
| **运维/行为** | 「凌晨」折叠依赖定时任务或「最后一次会话结束」后的延迟任务；若无固定凌晨 cron，需明确「何时算一天结束」。 |
| **隐私与数据** | 当日对话需被持久化或缓冲以供折叠，涉及留存策略与现有隐私模式（如 `sessionFlags.privacy`）的兼容。 |

### 3.4 必要性结论

- **功能必要性：高**。若产品目标包含「连续多日协作、用户自然说『继续昨天』」，则 5 天滑动情景记忆是**填补「语义 RAG」与「情景记忆」断层**的合理方案，与讨论中的认知设计一致。
- **实现必要性：取决于产品优先级**。若当前阶段不强调「跨日连贯」，可先保留为**设计储备**；若已收到「过夜就忘」的反馈，建议纳入路线图并做**分阶段实现**（见下）。

## 四、纳入项目建议

### 4.1 建议结论：**纳入为正式设计，分阶段实现**

- **纳入**：将「5 天滑动情景记忆 + 睡前折叠」列为项目**正式设计**，与现有 RAG/复盘/记忆层文档并列，便于后续工单拆解与实现顺序统一。
- **不跳过设计直接编码**：因涉及存储形态、触发时机、多会话/workspace 边界、与复盘/隐私的衔接，建议先落成**简短设计文档**（可基于本文档扩展），再拆工单。

### 4.2 建议的设计文档范围

1. **数据结构**：`rolling_ledger` 的 schema、存储路径（如 `workspace/.rzeclaw/memory/rolling_ledger.json`）、版本与兼容。
2. **折叠流水线**：触发条件（定时 vs 会话结束）、「当日输入」来源（今日缓冲 vs 当日持久化消息）、架构师（或专用折叠任务）的 Prompt 约定、与 `runRetrospective` 的先后/并行关系。
3. **淘汰策略**：Day -5 挤出时的「入 RAG vs GC」判定规则或接口。
4. **注入策略**：谁在何时读取账本、如何格式化为自然语言、与 `sessionSummary` 的合并方式、per-workspace vs per-session 的选取。
5. **配置与开关**：如 `memory.rollingLedger.enabled`、`memory.rollingLedger.windowDays`、折叠 cron 或等效配置。
6. **隐私与合规**：与 `privacy` 模式、审计日志的衔接。

### 4.3 建议的实现阶段（高 level）

| 阶段 | 内容 | 依赖 |
|------|------|------|
| **Phase A** | 账本存储与读写 API、schema 固化；无折叠、可先手写或占位数据验证「注入」效果。 | 无 |
| **Phase B** | 请求侧注入：Gateway/executor 在构造 system prompt 时读取账本并拼接。 | Phase A |
| **Phase C** | 「当日」数据来源与缓冲：在 flushToL1 / 会话结束时写入「今日」缓冲（或等价）。 | 现有 memory/gateway |
| **Phase D** | 折叠任务：定时或触发时，将今日缓冲压缩为日级摘要 + pending_tasks，更新账本并 FIFO 淘汰。 | Phase A、C |
| **Phase E** | 淘汰→RAG：Day -5 挤出时调用现有 L1/内源 RAG 写入（可选）。 | Phase D、现有 store/RAG |

### 4.4 与既有文档的衔接

- **RAG与复盘机制详细设计**：将「记忆折叠」作为「动态上下文（短期）」的**结构化、跨日扩展**，在「内源 RAG」或「复盘」章节增加对 rolling ledger 与折叠触发时机的引用。
- **MASTER_IMPLEMENTATION_PLAN**：可将本能力列为**后续 Phase**（如 Phase 17 或复盘增强子项），指向本文档及后续详细设计文档与工单。

## 五、总结

| 项目 | 结论 |
|------|------|
| **讨论内容** | 三级记忆（瞬时 / 5 天滑动情景 / 永久 RAG）+ 睡前折叠 + 清晨注入，能有效缓解「跨日失忆」并控制 token。 |
| **当前缺口** | 缺 5 天账本存储、夜间折叠流水线、淘汰→RAG、请求侧账本注入及「当日」数据来源的约定。 |
| **必要性** | 对「连续多日协作」场景**功能必要性高**；实现优先级依产品是否强调跨日连贯而定。 |
| **纳入建议** | **纳入**为正式设计；先补充设计文档与配置/隐私约定，再按 Phase A→E 分阶段实现，并与 RAG/复盘文档衔接。 |
