# 内嵌小模型选型与可行性讨论

本文档在**不修改现有设计与实现**的前提下，基于《内嵌模型+底层模型迭代处理》与行为树/状态机设计，讨论：**在无云端或云端不可用时的“快思考”内嵌小模型**的可行性，并推荐**开源、免费、可本地流畅运行**的模型选项。目标读者为决策与选型，不做具体实现与构建。

---

## 一、与现有设计的对齐

### 1.1 内嵌模型在架构中的角色

根据《内嵌模型+底层模型迭代处理》与 `BEHAVIOR_TREE_AND_STATE_MACHINE_DESIGN.md`：

| 角色 | 职责 | 与内嵌小模型的关系 |
|------|------|---------------------|
| **快思考 / 小脑** | 意图路由、状态分发、高置信度时直接走 BT/FSM，零 Token、低延迟 | **内嵌小模型即此角色**：在无云端或 API 不足/网络异常时，仍能对已构建的完整执行链路做「是否匹配某 flow」「选哪个 flow」「是否 ESCALATE_TO_CLOUD」等判断。 |
| **身体与经验** | 行为树/状态机 + 已固化的 Skill/flow 提供稳定执行；经验通过 RAG/微调迭代 | 内嵌模型 + 向量库实现「肌肉记忆」：历史 Skill/成功案例向量化，小模型在决策前检索到相关经验，即可路由到已有 flow，实现「大脑丢失，身体仍有执行能力」。 |
| **非大脑算力** | 部分判断与路由不交给云端 LLM，由本地完成 | 意图分类、会话级 FSM 的状态/迁移判断（若采用）、以及 BT 左侧「本能」是否命中的预判，均可由内嵌小模型承担，减轻云端与 Token 消耗。 |

因此：**内嵌小模型是“快思考”与“无云端时的基础能力”的承载体**，需满足：

- **本地可运行**：无网或弱网、API 限流时仍可用。
- **低资源**：常规/低配电脑（含仅 CPU、4–8GB 内存）能流畅跑。
- **结构化输出**：能稳定输出枚举或固定 JSON（如 `state`、`flowId`、`confidence`），便于与 Router/Executor/会话 FSM 对接。
- **可选扩展**：与向量模型配合做 RAG「软进化」、与 LoRA 等做「硬进化」时，有开源生态支持。

### 1.2 设计文档中的约束

- **intentClassifier**（主设计 §4.1、§10；对比文档 §2.1）：可选「本地小模型意图分类」，高置信度匹配 flow 时走 Executor，否则走 Agent。
- **会话级 FSM**（对比文档 §2.2）：若采用，本地小模型可只输出严格 JSON/枚举（如 `ROUTE_TO_LOCAL_SCRIPT` / `ESCALATE_TO_CLOUD`），仅 ESCALATE 时唤醒云端。
- **BEHAVIOR_TREE_CONTEXT_AND_SKILL_DESIGN.md §2.5**：本地小模型作 FSM 条件仲裁者，**只输出严格 JSON/Enum**，零成本高频轮询。

故选型时需优先考虑：**意图分类/路由 + 固定 schema 的 JSON 输出** 能力，而非开放域长文本生成。

## 二、可行性判断

### 2.1 技术可行性：结论为可行

- **任务形态匹配**：内嵌模型承担的是「分类/路由/状态枚举」，属于**限定输出空间**的任务。小模型（1.5B–4B）在 intent classification、slot filling、简单推理上已有成熟实践，且比开放域生成更容易控制格式（JSON/枚举）。
- **资源可行性**：当前开源小模型（Qwen2.5 1.5B/3B、Phi-3-mini、SmolLM、TinyAgent 等）在 GGUF 量化后可在 **纯 CPU、8GB 内存、无独显** 环境下运行；Ollama/llama.cpp 等工具链成熟，与“低配电脑也可跑”的目标一致。
- **与双轨进化的兼容**：  
  - **软进化**：向量模型（如 BGE-M3、Nomic-Embed）将 Skill/成功案例嵌入，小模型推理前做 RAG，检索到的经验作为上下文，无需改权重即可“学会”新路由。  
  - **硬进化**：Good/Bad case 收集 + 夜间 LoRA 微调，小模型权重迭代，与文档中描述一致；Qwen/Phi/SmolLM 等均有 LoRA/Unsloth 等生态支持。

### 2.2 与行为树/状态机设计的契合度

- **路由层**：Router 先规则，再可选小模型；小模型输出 `flowId` + `confidence`，Gateway 按阈值决定走 Executor 还是 runAgentLoop，与主设计完全兼容。
- **会话 FSM（若上）**：小模型输出 `state` 枚举，驱动 Idle / Local_Intercept / Executing_Task / Deep_Reasoning 迁移，不依赖云端，与讨论文档一致。
- **黑板/节点级记忆**：小模型仅消费「当前请求 + 可选 RAG 检索结果」；不要求小模型维护长会话，与「去中心化上下文、节点级记忆」无冲突。

**结论**：在现有设计框架下，**引入内嵌小模型做快思考与无云端兜底，在技术与设计上均可行**，无需为“内嵌模型”单独做大框架修改。

## 三、推荐的开源本地模型（仅讨论与推荐）

以下均为**开源、可商用或研究友好、支持本地部署**的模型；按「意图路由/快思考」与「向量嵌入」分两类。推荐优先级考虑：资源需求、JSON/工具调用能力、社区与工具链。

### 3.1 意图路由与“快思考”模型（主内嵌模型）

| 模型 | 规模 | 资源需求（约） | JSON/工具调用 | 说明与推荐理由 |
|------|------|----------------|---------------|----------------|
| **Qwen2.5 (1.5B / 3B Instruct)** | 1.5B / 3B | 2–3GB RAM（Q2/Q4 量化）、纯 CPU 可跑；4 核 CPU + 8GB 内存即可 | 指令遵循与结构化输出好；有 function calling 方向微调 | 文档已推荐；工具调用与指令遵循在同规模中突出；Ollama/GGUF 支持全，适合做**首选**意图路由 + 状态枚举。 |
| **Phi-3-mini (3.8B)** | 3.8B | 约 2.5–4GB RAM（量化）；CPU 可跑 | 有专门 **function calling** 微调版（如 HuggingFace 上 Phi-3-mini-instruct-function-calling）；JSON 输出在 2024 更新后显著提升 | 微软端侧优化，逻辑与结构化输出强；适合做「本能节点判断」与路由。 |
| **SmolLM2 / SmolLM3 (2B / 3B)** | 2B / 3B | 显存/内存需求低；支持 128k 上下文（SmolLM3）；CPU/GPU 均可 | 通用指令遵循；结构化输出需靠 prompt/后处理约束 | Hugging Face 开源，同参数量下性能优于部分 3B 基线；体积小、易部署，适合**低配首选**。 |
| **TinyAgent-1.1B** | 1.1B | 极低；为边缘/CPU 设计 | **专为 function calling / 任务路由** 设计；与 ToolRAG 结合时工具调用成功率高 | 若场景**极度强调“路由/工具调用”且资源非常紧张**，可作专门路由器；生态略小于 Qwen/Phi。 |
| **Qwen2.5-0.5B** | 0.5B | 极低 | 有用于 tiny-llm-agent 等轻量场景 | 仅在对延迟/内存有极端要求时考虑；能力上限明显低于 1.5B+。 |
| **MiniCPM 系列 (0.5B–4B)** | 0.5B–8B | 0.5B/1B 极低；4B 约 4GB 级 | 4B 有 function call、代码解释等；0.5B 偏极简 | 端侧定位明确；若希望「同一家族覆盖从极低配到 4B」可考虑。 |

**推荐组合（兼顾通用与低配）**：

- **首选**：**Qwen2.5-3B**（或 1.5B）作主内嵌“快思考”模型——与文档一致、工具调用与指令遵循好、Ollama/GGUF 生态完善；低配机可用 1.5B 或更强量化（如 Q2K）降低内存。
- **备选 / 低配优先**：**SmolLM2/3-3B** 或 **Phi-3-mini-3.8B**——前者更小、多语言与长上下文；后者逻辑与 function calling 强。
- **极简路由专用**：**TinyAgent-1.1B**——仅当明确只做「意图→flowId/state」且机器非常受限时作为专门路由模型。

### 3.2 向量嵌入模型（用于 RAG“软进化”）

与《内嵌模型+底层模型迭代处理》中“经验入库 → 向量检索 → 小模型带上下文决策”对应：

| 模型 | 规模/资源 | 说明 |
|------|-----------|------|
| **BGE-M3** | 约 569M 参数，多语言、多粒度、8192 token；2GB+ 级 | 文档已推荐；适合将 Skill 描述、成功案例嵌入，做混合检索；本地 CPU 可跑。 |
| **Nomic-Embed (v2)** | 针对 CPU/本地优化，MoE 设计 | 长上下文、本地推理友好（如 GPT4All）；与“低配也可跑”一致。 |

两者均为开源、可本地部署；若先做「快思考」再补「软进化」，可延后选型，不影响内嵌小模型本身的可行性。

### 3.3 运行方式与工具链（不涉及项目内实现）

- **Ollama**：一条命令拉取与运行 Qwen/Phi/SmolLM 等，适合快速验证与统一 API。
- **GGUF + llama.cpp**：量化档位多（Q2K–Q8），便于在 4GB/8GB 机器上做 CPU 推理。
- **Llamafile**：单文件、免安装，适合 Qwen2.5-1.5B 等极简部署与演示。

项目侧只需通过 HTTP/API 调用本地推理服务（如 Ollama），**无需在项目内实现模型加载与推理**，可行性高。

## 四、风险与边界（讨论层面）

- **准确率上限**：小模型在复杂/模糊意图上不如云端大模型，需通过「置信度阈值 + 低置信度走 Agent」保证体验；设计文档已覆盖。
- **多语言与中文**：Qwen2.5、BGE-M3 对中文友好；Phi/SmolLM 英文更强，中文需在验证阶段重点测。
- **依赖与运维**：内嵌模型依赖本地 Ollama/llama.cpp 等；需在部署文档中说明「可选组件」，避免无 GPU 用户误以为必须装大模型。

以上均为选型与运维层面的讨论，不改变现有 Gateway/Router/Executor 的接口与主流程设计。

---

## 五、小结与建议

- **可行性**：在现有行为树/状态机与「快思考 + 双轨进化」设计下，**内嵌小模型在技术与设计上均可行**；无需为“内嵌模型”做大框架修改。
- **推荐优先选型**：  
  - **意图路由/快思考**：**Qwen2.5-1.5B 或 3B** 作为主推荐；**SmolLM3-3B** 或 **Phi-3-mini-3.8B** 作为备选；**TinyAgent-1.1B** 仅在极度资源受限且只做路由时考虑。  
  - **向量模型（RAG 软进化）**：**BGE-M3** 或 **Nomic-Embed**，与文档一致，可后续接入。
- **下一步（仍为讨论/规划）**：确定是否采用会话级 FSM、intentClassifier 的 schema（如 `state` / `flowId` / `confidence`）与阈值策略；再在实现阶段选定单一主模型（如 Qwen2.5-3B）做 PoC，并约定「无云端时仅使用内嵌模型 + BT/FSM」的降级策略。

## 六、内嵌模型接入方式：随项目分发 vs 接口挂载（仅讨论）

在「内嵌小模型」可行且选型有方向的前提下，存在两种接入思路：**项目内随包提供模型（用户按性能选）** vs **项目保持纯净，仅提供挂载本地模型的接口（用户自备 Ollama/LLM Studio 等）**。下面从分发、体验、合规与维护四方面做纯讨论评估。

### 6.1 方案 A：项目内随包分发模型（用户按性能选择）

**含义**：项目或安装包内预置若干已下载好的模型（如 Qwen2.5-1.5B/3B 的 GGUF），用户首次使用或安装时根据本机性能（内存/显存）选择要启用的模型，系统直接加载使用。

| 维度 | 评估 |
|------|------|
| **分发与体积** | 单模型 GGUF 量化后约 1–4GB；若预置 2–3 档（1.5B/3B 等），仓库或安装包体积会显著增大，clone/下载、CI、镜像构建都会变慢；移动/复制项目成本高。 |
| **用户体验** | 优点：开箱即用，不要求用户会装 Ollama 或懂模型。缺点：占磁盘与内存；若用户已有自建 Ollama 且偏好其他模型，易产生「重复一份模型」或「不知用哪一份」的困惑。 |
| **合规与许可** | 需逐模型核对再分发条款（Qwen、Phi、SmolLM 等各自不同）；部分协议允许再分发但要求保留 license 与出处，部分对商用/再分发有单独约定；项目方需长期承担合规审查与版本更新时的复查。 |
| **维护** | 模型版本升级（安全补丁、量化格式）需在项目内跟进、测试、发版；与上游模型发布节奏强绑定，维护负担重。 |

### 6.2 方案 B：项目保持纯净，提供挂载本地模型的接口

**含义**：项目不包含任何模型文件；在配置或文档中约定「本地推理服务」的接口（如 OpenAI 兼容的 baseURL + model 名，或 Ollama / LLM Studio 等提供的 API）。用户自行用 Ollama、LLM Studio、llama.cpp 等下载、管理、运行模型，本项目只通过 HTTP/API 调用。

| 维度 | 评估 |
|------|------|
| **分发与体积** | 项目仓库与安装包保持小、纯净；无模型二进制，无合规再分发风险。 |
| **用户体验** | 需要用户多一步：安装 Ollama（或同类工具）并拉取所需模型。文档可提供「推荐模型列表 + 一行命令示例」（如 `ollama run qwen2.5:3b`），将门槛控制在可接受范围；熟悉本地 LLM 的用户可自由换模型、换量化档位。 |
| **合规与许可** | 不涉及模型再分发，合规责任在用户与模型提供方；项目仅调用本地 API，与调用云端 API 在合规上类似，负担最小。 |
| **维护** | 项目只维护「接口契约」（URL、请求/响应格式、可选 model 名）；模型版本与安全由 Ollama/上游生态负责，项目不随模型发版。 |

### 6.3 与设计定位的契合

- **主设计**中内嵌小模型是**可选**能力（intentClassifier 可选、无云端时兜底）；并非所有用户都会启用。若随包带模型，不用内嵌能力的用户也会被动承担体积与依赖；若为接口挂载，不用则不配置即可，与「可选」一致。
- **「根据性能选择模型」**在方案 B 下同样成立：文档中写明「低配推荐 qwen2.5:1.5b / SmolLM2，高配可用 qwen2.5:3b 或 Phi-3-mini」，用户在自己机器上拉取对应模型即可；无需项目预置多档模型。

### 6.4 结论（仅讨论）

- **更推荐方案 B（项目纯净 + 挂载接口）**：  
  - 仓库与安装包小、无模型再分发与合规压力、维护边界清晰；  
  - 用户按需选用 Ollama/LLM Studio 等成熟工具，按文档推荐按性能选模型，灵活性高；  
  - 与「内嵌模型为可选能力」的设计一致，不用的用户零负担。  
- **方案 A（随包分发模型）** 仅在强需求「完全离线、零额外安装、开箱即用」且能接受体积与合规维护成本时再考虑；若做，更现实的是提供**可选安装脚本/文档**，在用户主动执行时从官方源下载到用户指定目录，而非把模型二进制放进项目仓库或默认安装包。

## 七、本地模型统一接口与配置设计（仅设计，不实现）

本节在明确选择 **方案 B（项目纯净 + 挂载接口）** 的前提下，给出一套**统一本地模型接口**与**配置层设计**，用于：

- 为 **意图路由 / 会话 FSM / 行为树内本能仲裁** 提供统一的本地小模型调用入口；
- 为 **向量嵌入（RAG “软进化”）** 提供统一的嵌入服务抽象；
- 保证：项目本身**不分发任何模型文件**，仅通过 HTTP/API 与用户自备的本地服务（Ollama / LLM Studio / 自建 OpenAI 兼容服务等）通讯。

### 7.1 统一接口设计目标

- **抽象一致**：对上层（Gateway / Router / BT/FSM / 自主性模块）暴露的只有两类能力：  
  - 「**本地小模型推理**」：完成意图分类、状态枚举、轻量推理兜底。  
  - 「**本地嵌入服务**」：完成文本向量化，用于 RAG 与软进化。  
- **实现可插拔**：底层可以是 Ollama、OpenAI 兼容 HTTP、llama.cpp 包装、LLM Studio 提供的 REST 服务等，对上统一为少量接口与配置枚举。
- **配置集中**：统一在 config 中描述本地模型与嵌入配置，便于后续提供 UI/终端配置向导与「勾选开关」。
- **不引入实现细节**：本节只定义接口与配置 schema，不定义具体代码或模块名。

### 7.2 配置总览：localModel 与 vectorEmbedding

在现有总配置（如 `rzeclaw.json`）中，引入两个顶级或 namespaced 块（形式示意）：

```json
{
  "localModel": {
    "enabled": true,
    "provider": "ollama",          // "none" | "ollama" | "openai-compatible" | "custom-http"
    "endpoint": "http://127.0.0.1:11434",
    "model": "qwen2.5:3b",
    "apiKey": null,                // openai-compatible 时可选
    "timeoutMs": 5000,
    "modes": {
      "intentClassifier": {
        "enabled": true,
        "outputSchema": "router_v1",   // 预置 schema 名称，详见 7.3
        "confidenceThreshold": 0.7
      },
      "localReasoningFallback": {
        "enabled": false,             // 预留：无云端时轻量推理兜底
        "maxTokens": 256
      }
    }
  },
  "vectorEmbedding": {
    "enabled": false,
    "provider": "local-embed",        // "none" | "local-embed" | "openai-compatible" | "custom-http"
    "endpoint": "http://127.0.0.1:11434",
    "model": "bge-m3",
    "indexStoragePath": ".rzeclaw/embeddings",
    "collections": {
      "skills": { "enabled": true },
      "flows": { "enabled": true },
      "logs": { "enabled": false }
    }
  }
}
```

> 说明：上例仅为**设计级 schema 示意**，具体 key 名称、层级、默认值以主配置文档为准。本节不要求实际实现。

### 7.3 统一的意图分类输出 schema（router_v1）

为了让 Router、会话 FSM、BT/FSM 能以统一方式消费本地小模型输出，定义一个**固定 schema**（可命名为 `router_v1`），本地小模型的 prompt 与解析逻辑都围绕该 schema 约束：

```json
{
  "state": "ROUTE_TO_LOCAL_FLOW",  // 或 "ESCALATE_TO_CLOUD" | "NO_ACTION" | "UNKNOWN"
  "flowId": "deploy_prod",         // 可选：当 state = ROUTE_TO_LOCAL_FLOW 时必须
  "confidence": 0.82,              // 0~1
  "reason": "用户在请求部署生产环境"
}
```

- **state 枚举**（示例）：  
  - `ROUTE_TO_LOCAL_FLOW`：认为可由某 BT/FSM flow 处理。  
  - `ESCALATE_TO_CLOUD`：需要云端大模型（Deep_Reasoning）。  
  - `NO_ACTION`：无需动作（如闲聊、控制命令已经被第一层规则处理）。  
  - `UNKNOWN`：无法判断（视为低置信度）。  
- **flowId**：与行为树/状态机流程库中的 `id` 对应，用于直接调用 `Executor.run(flowId, params)`。  
- **confidence**：供 Router 或会话 FSM 与配置中的 `confidenceThreshold` 比较，决定是否采用该建议。  
- **reason**：可选，仅用于日志与调试，不参与逻辑判断。

在 `BEHAVIOR_TREE_CONTEXT_AND_SKILL_DESIGN.md` 中曾建议类似 schema（state + confidence）；本节将其**正式化为 router_v1**，并通过 `localModel.modes.intentClassifier.outputSchema` 进行引用，保证未来若有 router_v2 等升级时可兼容。

### 7.4 Router / 会话 FSM 与本地模型接口的关系

- **Router.match**：  
  1. 先跑现有规则（关键词、task_hint、正则等）；  
  2. 若规则未高置信度命中，且 `localModel.modes.intentClassifier.enabled = true`：  
     - 调用本地模型统一接口，得到 `router_v1` JSON；  
     - 若 `state = ROUTE_TO_LOCAL_FLOW` 且 `confidence ≥ confidenceThreshold`：按 `flowId` 调用 Executor；  
     - 否则走现有 `runAgentLoop`（或按 state/置信度进一步细分）。  
- **会话 FSM（若采用）**：  
  - 将 `router_v1.state` 作为「宏观状态迁移」输入（如 Idle → Local_Intercept / Deep_Reasoning），FSM 不直接感知底层是哪个模型或 provider。  
- **行为树 / 状态机内部本能仲裁**：  
  - 某些 Condition 节点可选择性调用本地模型接口，并按 `state`/`confidence` 决定走哪个分支，但仍建议大部分条件为**纯规则**，本地模型主要用于「入口路由」与「会话级宏观决策」。

### 7.5 向量嵌入（RAG “软进化”）接口与开关

结合第 3.2 节推荐的 BGE-M3 / Nomic-Embed，本节定义**统一嵌入接口与开关**，用于：

- 将 Skill 描述、flow 元信息、成功/失败案例等嵌入到本地向量索引中；
- 在路由或经验迭代时，对当前请求或上下文做向量检索，为本地小模型与 LLM 提供「经验候选」；
- 允许用户在配置或 UI 中**显式开启/关闭**该能力。

设计要点：

- **统一接口**：上层只调用诸如 `embed(text[]) -> float[][]` 与 `search(collection, query, topK) -> hits[]` 这样的抽象；底层通过 `vectorEmbedding.provider/endpoint/model` 决定具体实现。  
- **集合维度**：通过 `vectorEmbedding.collections` 声明哪些集合参与嵌入与检索（如 `skills`、`flows`、`logs`），便于用户只启用自己在意的部分。  
- **开关粒度**：  
  - 总开关：`vectorEmbedding.enabled`。  
  - 每集合开关：`vectorEmbedding.collections.<name>.enabled`。  
  - 未来可在 UI/配置向导中以「勾选」方式呈现。  
- **与双轨进化的关系**：  
  - 软进化：开启嵌入后，新成功案例入库、检索 → 本地小模型与 LLM 共享经验候选，但**不改变模型权重**。  
  - 硬进化：夜间 LoRA 等仍通过独立 pipeline，RAG 仅作为数据源之一。

### 7.6 用户体验与设置项（设计级）

在终端/配置向导/未来 UI 中，可围绕上述配置提供清晰的设置项（示例）：

- 「**启用本地快思考模型**」：勾选后显示 provider/endpoint/model 选择；未勾选则完全禁用本地意图分类。  
- 「**本地模型提供者**」：下拉选择 Ollama / OpenAI 兼容 / 自定义 HTTP，并提供相应示例（如 `ollama serve` + `ollama run qwen2.5:3b`）。  
- 「**RAG 软进化**」：总开关 + 每集合勾选，如「技能经验库」「流程经验库」「日志经验库」。  
- 所有选项只作用于**配置文件**，与本设计保持一致：**仅定义行为，不做实现**。
