# RAG 设计总结与智能体对接讨论

本文档基于 **docs/RAG相关内容** 及此前三份设计对话（行为树/状态机 [425dc2c7]、智能体总结 [7ab8a01b]、内嵌小模型与统一接口 [084d0317]），**仅做总结与讨论**：提炼 RAG 相关设计体现、可行性判断、与智能体/行为树/内嵌模型的对接要点与注意事项。**不包含新设计或实现。**

**详细设计**：可落地的架构、数据结构、配置、遥测与复盘流程见 **`RAG与复盘机制详细设计.md`**。

## 一、文档中「RAG」相关设计的体现（总结）

### 1.1 双轨 RAG 架构（核心划分）

| 轨道 | 定位 | 存储内容 | 设计意图 |
|------|------|----------|----------|
| **内源性 RAG** | 系统进化与「肌肉记忆」 | 动态上下文（短期黑板、近期对话摘要）、进化技能库（Skill/行为树 JSON、本地脚本）、路由肌肉记忆（意图识别高频命中记录） | 系统「越用越聪明」的底座；用户拥有**绝对控制权**，可查看、修改、维护。 |
| **外源性 RAG** | 领域知识与「外部脑」 | 开发文档、API 手册、世界观/设定集、规则书、垂直领域资料等 | 为 Agent 提供专业知识；支撑**垂类智能体**与多 Agent 的**知识隔离**。 |

- **类比**：内源 = 「程序性记忆（怎么做）」；外源 = 「语义性记忆（是什么）」。
- **物理与逻辑分离**：二者在存储、命名空间、检索入口上应严格区分，避免「引擎配置与玩家存档」混在「世界观设定」同一桶里。

### 1.2 内源性 RAG 的「绝对开放性」

- **拒绝黑盒**：不以不可读的二进制库为主；底层为**纯文本 + 本地向量库（如 ChromaDB）映射**，用户可审计、编辑。
- **配置即代码**：例如 `~/.ai_system/skills/` 下每个技能对应 `.json` + `.py`；用户可直接改 JSON（如调激进节点），保存即生效。
- **热重载**：用户修改后，向量库重新 Embedding 索引；下次路由/检索即使用人工修正后的「肌肉记忆」。

### 1.3 动机 RAG（Motivation / Intent RAG）

- **定位**：内源 RAG 的一部分，部署在**系统最前端**（紧贴 Gateway 或本地路由器）。
- **作用**：建立「人类模糊自然语言 → 系统标准机器指令（Event JSON）」的**翻译字典**，消除「提示词税」。
- **流程**：  
  1. 用户模糊输入（如「帮我随便弄点能跑的战斗框架」）；  
  2. **动机检索**（0 Token、毫秒级）：向量相似度匹配历史「动机簇」；  
  3. 命中则**直接重写为标准事件**投递 Event Bus（如 `{"intent": "generate_prototype", "target": "combat_system"}`），**不经过大模型**；  
  4. 未命中则升级给 LLM 澄清，用户确认后**固化写入动机 RAG**，形成可复用规则（含 `motivation_cluster`、`translated_events`、`context_requirement` 等）。
- **进化**：新动机经 LLM 澄清 + 用户确认后打包写入，下次类似说法即可零 Token 触发。

### 1.4 外源性 RAG 与多 Agent / 垂类

- **命名空间（Namespaces / Collections）**：外源 RAG 按**集合**划分，不是单一杂烩。
- **Agent 挂载**：例如「代码 Agent」只挂载「游戏开发与算法」命名空间；「叙事 Agent」挂载「战锤设定」「科幻世界观」。这样检索时**知识隔离**，避免代码 API 污染叙事、叙事设定污染代码。
- **与垂类智能体**：未来不同垂类 Agent 各绑定不同外源 RAG 集合，实现专业化。

### 1.5 双轨 RAG 的协同流程（文档示例）

1. **内源检索（找技能）**：本地小模型/路由通过向量检索内源 RAG，发现有无现成 Skill/flow（如「通用数据提取」）。
2. **派发与挂载**：将任务派发给对应 Agent（或 flow），并附带匹配到的行为树/技能。
3. **外源检索（找知识）**：Agent 执行行为树时若缺领域知识（如「战锤棋子属性格式」），触发外源 RAG，从用户灌入的规则书等提取。
4. **执行与反哺**：大模型结合外源知识完成任务；成功后可将此次执行路径提炼为新 Skill 写入内源 RAG，完成**自我进化**。

### 1.6 复盘机制（Retrospective Engine）

- **架构定位**：**系统梦境（System Dreaming）**——离线的、定时或触发的异步工作流（如凌晨空闲或用户 `/启动系统复盘`），**不放在主请求链路**。
- **执行主体**：**架构师 Agent（Meta-Agent）**——仅负责监控与修改系统本身的配置/代码/JSON，不处理业务任务。
- **数据来源**：Event Bus 运转时，将事件流水（成功/失败、耗时、Token）异步写入**遥测日志库（Telemetry Log）**，供复盘消费。
- **针对 RAG 的复盘**：  
  - **动机 RAG**：意图重叠修剪（如「改 Bug」与「排查代码错误」合并为更泛化规则）。  
  - **外源/内源 RAG**：检索未命中分析，生成《知识库缺失报告》，提示用户补充资料。
- **针对 BT/FSM 的复盘**：节点耗时与失败率统计；反复失败节点可触发自动重构（如 LLM 重写脚本，沙盒通过后替换）；链路压缩/短路优化。
- **针对 Agent 的复盘**：职责越界审查（如运维 Agent 被误用于「图片转换」→ 修正路由表，改由美术 Agent 处理）。
- **安全**：**PR 审批流**——架构师只生成补丁/待审 JSON，不直接覆盖；次日**早报**列出变更，用户 Y/N 确认后才合并（Human-in-the-loop）。

## 二、与现有智能体 / 行为树 / 内嵌模型设计的对应关系

### 2.1 与《智能体设计总结与对接要点》的对应

| RAG 概念 | 智能体侧对应 | 说明 |
|----------|--------------|------|
| 内源 RAG（技能/路由肌肉记忆） | Agent 局部上下文、Skill 挂载、flow 库 | 《智能体相关》中 Agent 含「局部 RAG」；内源 RAG 的「技能库」即 BT/flow + Skill 脚本，与「肌肉执行层」一致。 |
| 动机 RAG（最前端） | Gateway + Router + 意图分类 | 动机 RAG 的「模糊→标准事件」与 Router + 本地小模型意图分类（router_v1）在**位置与目标**上一致；可理解为动机 RAG 是「向量化、可进化的意图层」，Router 规则 + intentClassifier 是其实施形态之一。 |
| 外源 RAG 命名空间 | 多 Agent 知识隔离、垂类 Agent | 若采用多 Agent 实体（路径 B），每个 Agent 挂载不同外源 RAG 集合，与文档中「代码 Agent / 叙事 Agent 各挂不同 RAG」一致。 |
| Event Bus | 事件流水 → 遥测日志 → 复盘 | 复盘依赖 Event Bus 上的事件与结果；与「Event Bus 为骨架、Gateway 为节点」一致，需约定事件 schema 含足够遥测字段（耗时、Token、节点 ID 等）。 |
| 架构师 Agent | 元/系统级 Agent | 当前设计为「单会话多 flow」，未显式引入多 Agent；架构师 Agent 可作为**特殊单例**（只做复盘与配置修改），与「业务 Agent」在权限与触发时机上隔离。 |

### 2.2 与行为树 / 状态机设计的对应

| RAG 概念 | BT/FSM 侧对应 | 说明 |
|----------|----------------|------|
| 内源 RAG（进化技能库） | flow 库、Skill 双形态（脚本型 + 流程型） | BEHAVIOR_TREE_CONTEXT_AND_SKILL_DESIGN：Skill = 固化行为树链路；进化产出新 flow/新节点写入流程库，即「内源 RAG 反哺」的数据来源。 |
| 路由肌肉记忆 | Router.match → flowId、意图分类 + confidence | 内源 RAG 中「意图识别高频命中」可与 Router 的规则 + 本地小模型输出对应；经验迭代（优选 flow/分支）即「肌肉记忆」的统计体现。 |
| 复盘 → BT 节点/脚本重构 | 进化安全、沙盒试飞、merge 前审批 | BT 文档与《智能体相关》均强调：写入流程库/脚本的变更需沙盒 + 测试 + 可选用户确认；与复盘的 PR 审批流一致，应**统一为同一套安全策略**。 |

### 2.3 与内嵌小模型 / 统一接口设计的对应

| RAG 概念 | 内嵌模型/向量侧对应 | 说明 |
|----------|----------------------|------|
| 内源 + 动机 RAG 的向量检索 | vectorEmbedding（RAG 软进化）、localModel.intentClassifier | 《内嵌小模型选型与可行性讨论》§7：vectorEmbedding 为「RAG 软进化」提供统一嵌入与检索；intentClassifier 输出 router_v1，可与「动机 RAG 命中后输出标准事件」对齐——即**动机 RAG 命中时相当于 intentClassifier 的高置信度 ROUTE_TO_LOCAL_FLOW**。 |
| 外源 RAG 的向量库 | vectorEmbedding.collections（如 skills / flows / 用户外接） | 当前设计为 collections 可配置（skills、flows、logs）；**外源 RAG** 可对应为**额外集合**（如 `external_docs`、`domain_knowledge`），由用户勾选与维护，与「用户外接 RAG」一致。 |
| 用户可维护、可勾选 | vectorEmbedding.enabled、collections.*.enabled | 内嵌文档已约定：项目实现向量嵌入与检索，但**用户通过配置/勾选**决定是否启用及哪些集合启用；与 RAG 文档中「绝对开放性、用户自主维护」一致。 |

## 三、可行性判断（讨论）

### 3.1 双轨划分（内源 vs 外源）

- **可行**：在存储与检索层面区分「系统自生的进化数据」与「用户外接的知识数据」，在工程与认知上均合理；与现有「flow 库 + Skill」与「知识库/记忆」的已有划分可兼容。
- **注意**：内源需保证「可读、可编辑、热重载」，避免纯黑盒向量库；外源需从配置上支持多集合/多命名空间，便于多 Agent 或垂类挂载。

### 3.2 动机 RAG

- **可行**：与当前「Router + 规则 + 可选本地小模型」在**目标与位置**上一致；差异在于动机 RAG 强调**向量化、可进化**的「模糊→标准事件」映射，而不仅是静态规则。
- **对接选择**：  
  - **路径 1**：将「动机 RAG」视为 Router 的**增强数据源**——先查动机 RAG（向量检索），命中则直接产出 `flowId` + 标准事件；未命中再走规则 + intentClassifier。  
  - **路径 2**：动机 RAG 与现有 router_v1 合并——intentClassifier 的 prompt 中注入「动机 RAG 检索结果」作为少样本或规则列表，输出仍为 router_v1 schema；固化时写入动机 RAG 的条目与 router_v1 的 state/flowId 对齐。  
- **建议**：在设计中**显式区分**「规则 + 意图分类」与「动机 RAG 检索」的调用顺序与数据流（例如先动机 RAG 再规则再 intentClassifier），避免两套「意图层」重复或冲突。

### 3.3 外源 RAG 与多 Agent / 垂类

- **可行**：外源 RAG 以集合/命名空间存在，Agent 或 flow 执行时按「当前 Agent 或 flow 绑定哪些集合」做检索，技术上为配置 + 检索时 filter。
- **与当前阶段**：若暂不引入多 Agent 实体（路径 A），可先实现**单会话下多外源集合**（如「代码」「叙事」两个集合），由 Router/flow 在执行时指定使用哪一集合；待多 Agent 实体引入后，再绑定到 Agent 蓝图。

### 3.4 复盘机制（系统梦境 + 架构师 Agent）

- **可行**：复盘作为**离线/定时任务**，不占用主链路；遥测日志格式需在设计阶段约定（含 flowId、nodeId、耗时、Token、成功/失败等），与 Event Bus 或 Gateway 的日志写入点对接。
- **安全**：必须**不自动覆盖**；仅生成补丁/待审变更，经早报 + 用户确认后合并，与 BT/智能体文档中的「进化安全」统一。
- **实现量**：架构师 Agent 可视为「只读遥测 + 调用 LLM 生成建议/补丁」的专用流程，与主业务 runAgentLoop 隔离；需单独定义其权限边界与可修改范围（仅限配置/JSON/脚本，不碰核心运行时）。

## 四、需要对接的设计要点与注意事项

### 4.1 术语与边界统一

| 概念 | 建议约定 |
|------|----------|
| **内源 RAG** | 系统在使用中产生的、与「进化」相关的数据：上下文摘要、Skill/flow、路由命中记录等；**用户可查看、修改、维护**，存储与索引对用户开放。 |
| **外源 RAG** | 用户外接的领域/垂直知识；可多集合、多命名空间；用于执行时的知识增强与垂类 Agent，**用户负责灌入与维护**。 |
| **动机 RAG** | 内源 RAG 的子集，专门存储「模糊意图 → 标准事件」的映射；与 Router、router_v1 在入口处对接，需明确与「规则 + intentClassifier」的调用顺序与数据格式。 |

### 4.2 与 Router、router_v1、会话 FSM 的对接

- **动机 RAG 输出**应与 router_v1 或 Executor 输入**兼容**：例如动机 RAG 命中时产出 `{ state: "ROUTE_TO_LOCAL_FLOW", flowId, params }` 或等价 Event JSON，以便直接进入 Executor 或 Event Bus，无需再经 LLM。
- **会话 FSM**：若采用会话级 FSM，动机 RAG 命中导致的「直接执行 flow」应对应 FSM 的 Local_Intercept 或 Executing_Task，在文档中写明**状态迁移与数据流**，避免与「未命中走 intentClassifier」路径产生两套状态语义。

### 4.3 与 vectorEmbedding、localModel 配置的对接

- **内源 RAG** 使用的向量存储与检索，应复用或对齐 **vectorEmbedding** 的接口与配置（embed、search、collections）；内源集合（如 skills、flows、motivation）与现有设计中的 collections 对应；**用户可勾选**哪些内源集合启用。
- **外源 RAG** 可对应 **额外 collections**（如 `external_docs`、`domain_*`），由用户配置 endpoint/model 与索引路径，与「项目实现、用户勾选与维护」一致。
- **localModel.intentClassifier** 与动机 RAG 的关系：建议在配置或设计文档中写明「动机 RAG 先查还是 intentClassifier 先调」及「二者结果如何合并」（例如动机高置信命中则不再调 intentClassifier）。

### 4.4 复盘与进化安全的一致性

- **BT/FSM 进化**（新 flow、新节点、脚本替换）与 **RAG 复盘**（动机合并、知识缺失报告、节点重构）应共用同一套**安全策略**：  
  - 所有对「流程库、Skill、动机 RAG、外源索引」的**写入**均经沙盒/测试 + 可选用户确认；  
  - 架构师 Agent 只生成补丁/报告，不直接写库；  
- 在 Phase 13 或复盘/进化相关工单中**显式引用**该策略，避免 BT 文档、智能体文档、RAG 文档各说各话。

### 4.5 文档与阶段建议

- 在 **BEHAVIOR_TREE_AND_STATE_MACHINE_DESIGN** 或 **MASTER_IMPLEMENTATION_PLAN** 中增加对 **RAG相关内容** 与本文档的**交叉引用**，说明：  
  - 内源/外源/动机 RAG 分别对应哪些现有模块（flow 库、Skill、知识库、Router、vectorEmbedding）；  
  - 复盘机制与 BT 进化、安全策略的衔接。  
- **实现阶段**：可先落实「内源 RAG（技能/flow 向量化 + 检索）」与「外源 RAG（多集合）」的存储与检索接口，再单独做「动机 RAG」与 Router 的集成，最后再上「复盘 + 架构师 Agent」，以降低首轮复杂度。

## 五、小结

- **总结**：RAG 文档中的设计体现为**双轨 RAG（内源进化 vs 外源知识）、动机 RAG（模糊→标准事件、最前端）、绝对开放性（可读可改热重载）、外源命名空间（多 Agent/垂类）、复盘机制（系统梦境 + 架构师 Agent + PR 审批）**；与智能体设计中的 Router、Agent 局部记忆、Event Bus、与行为树中的 flow/Skill、进化安全、与内嵌模型中的 vectorEmbedding、router_v1 均有明确对应点。
- **可行性**：双轨划分、动机 RAG、外源多集合、复盘机制在技术与现有设计上**均可行**；需在**动机 RAG 与 Router/intentClassifier 的先后与数据格式**、**复盘与 BT 进化安全策略统一**、**vectorEmbedding 与内源/外源集合的映射**上做显式对接。
- **对接要点**：内源/外源/动机 RAG 术语与边界；动机 RAG 与 router_v1、会话 FSM 的兼容；vectorEmbedding.collections 与内源/外源集合的对应；复盘与进化安全一致；文档交叉引用与分阶段实现顺序。
- **注意事项**：动机 RAG 与现有意图层二选一或明确调用顺序；用户可维护与可勾选贯穿内源与外源；架构师 Agent 权限与可修改范围需单独约定。

以上为**纯总结与讨论**，不包含新设计或实现；具体设计决策与实现以主计划与后续工单为准。  
**详细设计文档**：`RAG与复盘机制详细设计.md`（双轨 RAG 集合与存储、动机 RAG 调用顺序与 schema、向量层配置、复盘机制与遥测、安全策略、实现阶段建议）。
